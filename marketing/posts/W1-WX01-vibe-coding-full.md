## 正文

### 为什么写这篇

过去几个月，我用 Vibe Coding 的方式做了一个 macOS 语音输入工具——Xisper。从第一行代码到现在的 Alpha 版本，整个过程中 AI 深度参与了代码编写，而我负责的是架构设计、产品决策和质量把关。

这篇文章完整记录这个过程：为什么做、怎么做、遇到了什么、学到了什么。

---

### 起点：一个程序员的"输入焦虑"

我在语音 AI 行业工作，做客户端开发。每天和 ASR（自动语音识别）、转写技术打交道。

但说实话，市面上的语音输入工具我都不太满意。要么需要你去申请 API Key、配置服务商，门槛太高。要么识别准确率一般，碰到专业术语就翻车。还有一些走向了另一个极端——往输入法里塞 Agent、搜索、对话，越做越重。

我想要的很简单：**按一个键，说话，文字精准地出现在我正在用的 App 里。** 专业术语不出错，不需要折腾配置，用完就走。

市面上没有完全符合我预期的，那就自己做一个。

### 什么是 Vibe Coding

Vibe Coding 这个词最近很火，但我理解的 Vibe Coding 不是"让 AI 写完所有代码然后跑起来就行"。

我的定义是：**人负责设计和决策，AI 负责实现和编码，两者高频协作迭代。**

具体来说，我的工作流是这样的：

1. **我先想清楚架构**——模块怎么划分、数据怎么流转、接口怎么设计。这一步完全靠自己，AI 给不了你全局视角。

2. **用自然语言描述需求给 AI**——打开 Cursor，用人话说"我需要一个什么什么功能"，AI 来写代码。

3. **Review + 修改**——我像做 Code Review 一样审查 AI 的产出。大概 60% 的情况需要修改：过度设计、错误处理不足、命名不合理、架构偏离我的预期。

4. **迭代到满意**——来回几轮，直到代码符合我的标准。

> 📸 **[图 2]**
> **需要准备**：两张截图拼在一起。左边是你在 AI 工具中描述需求的截图（自然语言输入），右边是 AI 产出的代码。体现"需求是人话 → 产出是代码"的过程。

这种方式的效率提升是真实的——单个模块的开发速度确实快了很多。但**最终代码的质量上限取决于你的审美和标准，不取决于 AI 的初始输出。** 如果你自己不知道好的代码长什么样，AI 也救不了你。

### 技术选型

先说整体技术栈：

- **前端**：Electron + Vue 3 + TypeScript（TSX 为主）
- **后端**：Cloudflare Workers + Hono 框架
- **存储**：Cloudflare D1（数据库）+ R2（文件存储）+ KV（键值缓存）
- **ASR**：豆包（Volcengine）实时流式语音识别
- **AI 后处理**：DeepSeek（通过 Workers 代理）
- **原生模块**：Rust + napi-rs（Fn 键监听）
- **项目管理**：pnpm monorepo + Turborepo

为什么选 Electron？这个问题我被问过很多次。

简单说：我是客户端开发出身，Electron 是我最熟悉的跨平台方案，而且 Xisper 的核心不在 UI 性能，在于音频处理和系统级交互。对于这个场景，Electron 的生态和开发效率是合理的选择。

> 📸 **[图 3]**
> **需要准备**：项目的 monorepo 目录结构截图（终端 `tree` 或者 VS Code 文件树），展示 apps/desktop、apps/web、apps/services、packages/ 等目录。让技术读者看到项目的工程化程度。

为什么选 Cloudflare 而不是传统服务器？因为我是一个人在做这件事，我需要尽可能少的运维负担。Workers 的 Serverless 模式、D1 的 SQLite 兼容、R2 的免费存储额度，对独立开发者非常友好。后面会单独写一篇 Cloudflare 全家桶的实战体验。

### 核心架构：两阶段识别

Xisper 的核心技术决策是**两阶段识别架构**：

**第一阶段：ASR 语音识别**——用户按住 Fn 说话，音频实时流式传到后端，ASR 引擎把语音转成文字。

**第二阶段：AI 后处理**——拿到原始转写文本后，交给 AI（DeepSeek）做优化：修正口语、补标点、调整格式。如果用户选了行业角色，AI 还会根据行业的纠错规则修正专业术语。

> 📸 **[图 4]**
> **需要准备**：一张架构流程图。可以手画也可以用工具画，大致展示：用户说话 → 音频流 → ASR 识别 → 原始文本 → AI 后处理（+ 行业角色规则）→ 最终文本 → 粘贴到 App。不需要太精细，能看清流程就好。

为什么要两阶段而不是一步到位？因为 ASR 和 LLM 擅长的事情不一样。ASR 擅长把声音转成字，但它对上下文、专业术语的理解力有限。LLM 擅长理解语义和修正错误，但它不能直接处理音频。两者配合，取各自的长处。

### 行业角色：精准识别的核心

这是我最自豪的设计决策之一。

通用的语音识别碰到专业术语几乎一定会出错。"民事诉讼法"变成"民事素松法"，"二甲双胍"变成"二甲双瓜"。这不是 ASR 模型不好，而是通用模型没有这些领域的先验知识。

Xisper 的方案是"行业角色"：用户选一个行业（目前支持开发者、法律、医学、金融、教育、通用），系统会加载对应的专业词汇和纠错规则。AI 后处理阶段会根据这些规则修正转写结果。

为什么叫"行业角色"而不是"词库"或者"纠错包"？因为它不只是加了几个词，而是让 AI 进入了一个"角色"——理解这个行业的上下文、表达习惯和术语体系。

### 产品哲学：输入法只做输入

在做 Xisper 的过程中，我一直在思考一个问题：语音输入工具的边界在哪里？

很多产品在往输入法里加 Agent、搜索、对话助手。但我的判断是：**输入法是一个交互入口，它的下游千变万化。** 你不知道用户输入的文字会交给谁——可能是一封邮件的收件人、一段代码的编译器、一份合同的审查律师、甚至是另一个 AI。

既然你不知道下游是什么，那输入的产出就应该**尽可能精准**。精准是输入法的本分。

所以 Xisper 不做 Agent。不做搜索。不做对话。只做一件事：让你说出的每个字都准确地变成文字。

---

### Vibe Coding 的体会

用 Vibe Coding 做完一个完整产品后，我最大的体会是：

**AI 能写出任何一段代码。但"该不该写"、"该写在哪"、"该用什么模式"——这些决策是人的活。**

现在没有什么代码是 AI 写不了的。程序员真正不可替代的，是品位——你多年积累的抽象能力、设计模式的直觉、对技术方案的判断力。

Vibe Coding 不是让程序员失业，是让有品位的程序员更高效。
