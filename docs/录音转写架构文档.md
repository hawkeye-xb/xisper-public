# Xisper 录音转写架构文档

## 一、总体架构概览

### 1.1 核心模块组成

```
┌─────────────────────────────────────────────────────────────┐
│                    LiveTranscribeView                       │
│              (UI 组件 - 用户交互层)                         │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ├── 监听快捷键事件
                       ├── 管理 UI 状态
                       ├── 处理后处理流程
                       └── 管理数据库写入
                       │
┌──────────────────────┴──────────────────────────────────────┐
│              RecordingCoordinator                           │
│           (录音协调器 - 会话管理层)                         │
│  - 会话队列管理 (Session Queue)                            │
│  - 误触检测 (Mistouch Detection)                           │
│  - 场景匹配 (Scenario Matching)                            │
└──────────────────────┬──────────────────────────────────────┘
                       │
         ┌─────────────┼─────────────┐
         │             │             │
         ▼             ▼             ▼
  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
  │AudioRecorder│ │  WSManager  │ │Recognition  │
  │   (录音)    │ │ (WebSocket) │ │   Store     │
  └─────────────┘ └─────────────┘ └─────────────┘
         │             │             │
         │             │             │
         ▼             ▼             ▼
    麦克风采集      ASR 服务连接   状态/结果管理
```

### 1.2 数据流向

```
用户按键 → 快捷键触发 → Coordinator 创建会话 → 音频采集
                                                  ↓
数据库保存 ← UI 更新 ← 后处理(可选) ← ASR 识别 ← WebSocket 传输
```

## 二、录音启动流程

### 2.1 触发机制

系统支持三种录音模式:

1. **PRESS_HOLD** (按住说话): 按下开始,松开结束
2. **PRESS_TOGGLE** (按键切换): 第一次按下开始,第二次按下结束
3. **AUTO_VAD** (自动检测): 基于 VAD (Voice Activity Detection)

#### 流程图

```
┌────────────────────────────────────────────────────────────┐
│  用户按下快捷键                                            │
└──────────────┬─────────────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────────┐
│  Electron Hotkey Module 捕获事件                          │
│  - actionId: 'hold-to-record' / 'toggle-record'          │
│  - event: 'start' / 'stop'                               │
│  - timestamp: 事件时间戳                                  │
└──────────────┬───────────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────────┐
│  LiveTranscribeView.onMounted                             │
│  监听: electronAPI.hotkey.onActionTriggered()            │
└──────────────┬───────────────────────────────────────────┘
               │
               ▼
       ┌──────┴──────┐
       │  event type │
       └──────┬──────┘
              │
      ┌───────┴───────┐
      │               │
      ▼               ▼
   'start'         'stop'
      │               │
      ▼               ▼
coordinator      coordinator
.handleStart()   .handleStop()
```

### 2.2 启动详细流程

```
coordinator.handleStart()
    │
    ├─ 1. 创建新会话 (createSession)
    │   ├─ 生成唯一 sessionId: `session-${seq}-${timestamp}`
    │   ├─ 初始化会话状态: 'pending'
    │   ├─ 创建音频缓冲区: audioBuffer = []
    │   └─ 记录开始时间: startTime = Date.now()
    │
    ├─ 2. 判断是否可以立即启动
    │   │
    │   ├─ 场景 A: 无活动会话 (activeSession === null)
    │   │   └─► 直接启动 (startSession)
    │   │
    │   └─ 场景 B: 已有活动会话
    │       ├─ 检查队列是否已满 (MAX_PENDING_SESSIONS = 5)
    │       │   ├─ 已满 → 拒绝新会话,显示提示
    │       │   └─ 未满 → 加入队列
    │       │
    │       └─► 立即开始录音 (音频缓冲到队列中)
    │           └─ audioRecorder.startRecording()
```

### 2.3 会话启动 (startSession)

```
startSession(session)
    │
    ├─ 1. 更新状态
    │   ├─ coordinator.state = 'processing'
    │   ├─ session.state = 'connecting'
    │   └─ streamId++ (触发 LiveTranscribeView 的状态重置)
    │
    ├─ 2. 重置识别状态
    │   ├─ recognitionStore.reset()
    │   ├─ 清空已提交文本缓存
    │   └─ 清空场景匹配状态
    │
    ├─ 3. 建立 WebSocket 连接
    │   ├─ session.ws = createWSManager()
    │   ├─ 构建 ASR 配置 (buildASRConfig)
    │   │   ├─ appid, token (认证信息)
    │   │   ├─ resource_id (资源标识)
    │   │   ├─ 音频参数 (format, rate, bits, channel)
    │   │   ├─ 语言设置 (language: 'zh-CN')
    │   │   └─ workflow 配置 (音频处理流程)
    │   │
    │   └─► ws.connect(config) - 见"三、ASR 连接"
    │
    ├─ 4. 启动音频录制
    │   ├─ audioRecorder.startRecording()
    │   │   ├─ 请求麦克风权限
    │   │   ├─ 创建 AudioContext (sampleRate: 16000Hz)
    │   │   ├─ 创建音频处理节点
    │   │   └─ 开始采集音频数据
    │   │
    │   └─ session.state = 'recording'
    │
    └─ 5. 开始发送音频
        └─► startSendingAudioToWS(session)
            ├─ 先发送缓冲的音频 (如果有)
            └─ 启动定时器 (每 100ms 发送一次)
```

## 三、ASR 连接建立

### 3.1 WebSocket 连接流程

```
WSManager.connect(config)
    │
    ├─ 1. 构建 WebSocket URL
    │   └─ wss://openspeech.bytedance.com/api/v3/sauc/bigmodel_async
    │       ├─ api_app_key: config.appid
    │       ├─ api_access_key: config.token
    │       ├─ api_resource_id: 资源 ID
    │       └─ cluster: 集群标识 (可选)
    │
    ├─ 2. 创建 WebSocket 实例
    │   ├─ ws = new WebSocket(url)
    │   ├─ ws.binaryType = 'arraybuffer'
    │   └─ 设置事件处理器 (onopen, onmessage, onerror, onclose)
    │
    ├─ 3. 等待连接建立
    │   ├─ 超时时间: 10 秒 (CONNECTION_TIMEOUT_MS)
    │   └─ 轮询检查 ws.readyState === OPEN
    │
    ├─ 4. 发送初始配置请求 (Full Client Request)
    │   └─► createFullClientRequest(config)
    │       │
    │       ├─ 构建 JSON 配置
    │       │   ├─ app: { appid, token, cluster }
    │       │   ├─ user: { uid }
    │       │   ├─ audio: { format, rate, bits, channel, codec }
    │       │   └─ request: { workflow, language, options... }
    │       │
    │       ├─ 序列化 JSON → Uint8Array
    │       │
    │       ├─ 可选: Gzip 压缩
    │       │
    │       └─ 组装二进制协议包
    │           ├─ Header (4 bytes)
    │           │   ├─ version: 0x1
    │           │   ├─ headerSize: 0x1 (4字节)
    │           │   ├─ messageType: 0x0b (FullClientRequest)
    │           │   ├─ serializationMethod: 0x1 (JSON)
    │           │   └─ compressionMethod: 0x1 (Gzip) 或 0x0 (None)
    │           │
    │           ├─ PayloadSize (4 bytes, big-endian)
    │           │
    │           └─ Payload (JSON 数据)
    │
    └─ 5. 连接成功
        ├─ state = 'connected'
        ├─ 记录连接时间: connectionStartTime
        └─ 重置重连计数: reconnectAttempts = 0
```

### 3.2 二进制协议格式

#### 协议头结构 (4 bytes)

```
Byte 0: [version:4bit][headerSize:4bit]
Byte 1: [messageType:4bit][messageFlags:4bit]
Byte 2: [serializationMethod:4bit][compressionMethod:4bit]
Byte 3: reserved (0x00)
```

#### 消息类型 (MessageType)

- `0x0b`: FullClientRequest (初始配置请求)
- `0x01`: AudioOnlyClientRequest (纯音频数据)
- `0x09`: FullServerResponse (服务器完整响应)
- `0x0f`: ServerError (服务器错误)

#### 消息标志 (MessageFlags)

- `0x00`: None (普通消息)
- `0x01`: LastPackage (最后一个包)
- `0x08`: WithSequence (包含序列号)

#### 完整消息结构

```
┌────────────────────────────────────────────┐
│  Header (4 bytes)                          │
├────────────────────────────────────────────┤
│  Payload Size (4 bytes, big-endian)        │
├────────────────────────────────────────────┤
│  Payload (可选压缩)                        │
│  - JSON 配置 (FullClientRequest)           │
│  - 音频数据 (AudioOnlyClientRequest)       │
└────────────────────────────────────────────┘
```

### 3.3 重连机制

```
连接断开触发 (ws.onclose)
    │
    ├─ 检查重连条件
    │   ├─ NOT isManualClose (非手动关闭)
    │   ├─ NOT suppressReconnect (未抑制重连)
    │   ├─ state === 'connected' (之前已连接)
    │   └─ config 存在
    │
    ├─ 重连次数检查
    │   ├─ reconnectAttempts < 3 (MAX_ATTEMPTS)
    │   │   └─► 执行重连
    │   │       ├─ reconnectAttempts++
    │   │       ├─ 等待 1 秒 (RECONNECT_DELAY_MS)
    │   │       └─► 调用 connect(config)
    │   │
    │   └─ reconnectAttempts >= 3
    │       └─► 停止重连,报告错误
    │
    └─ 特殊情况: 发送 isLast 后不重连
        └─ suppressReconnect = true
```

## 四、音频数据采集与发送

### 4.1 音频采集流程

```
AudioRecorder.startRecording()
    │
    ├─ 1. 请求麦克风权限
    │   └─ navigator.mediaDevices.getUserMedia({
    │       audio: {
    │         echoCancellation: true,      // 回声消除
    │         noiseSuppression: true,      // 降噪
    │         autoGainControl: true,       // 自动增益
    │         channelCount: 1,             // 单声道
    │       }
    │     })
    │
    ├─ 2. 创建音频处理链
    │   ├─ AudioContext (sampleRate: 16000Hz)
    │   ├─ MediaStreamSource (麦克风输入)
    │   ├─ ScriptProcessor (音频处理节点, bufferSize: 4096)
    │   └─ Destination (音频输出)
    │
    ├─ 3. 音频处理回调 (onaudioprocess)
    │   │
    │   └─► 每次触发处理 4096 个采样点 (约 256ms @ 16kHz)
    │       │
    │       ├─ A. 获取音频数据
    │       │   └─ inputData = inputBuffer.getChannelData(0)
    │       │       └─ Float32Array (范围: -1.0 ~ 1.0)
    │       │
    │       ├─ B. 计算音量
    │       │   └─ calculateVolume(inputData)
    │       │       ├─ RMS (均方根) 计算
    │       │       ├─ Peak (峰值) 检测
    │       │       ├─ 混合算法: volume = rms*0.7 + peak*0.3
    │       │       └─► 回调: onVolumeChange(volume)
    │       │
    │       └─ C. 转换并缓冲音频
    │           └─ floatTo16BitPCM(inputData)
    │               ├─ Float32 → Int16 转换
    │               │   └─ int16[i] = float32[i] * 32767
    │               │
    │               └─ 加入缓冲区: audioBuffer.push(pcmData)
    │
    └─ 4. 定时发送缓冲音频
        └─ setInterval(() => sendBufferedAudio(), 100ms)
            │
            └─► sendBufferedAudio()
                ├─ 合并所有缓冲块 → Int16Array
                ├─ 转换为 Uint8Array (little-endian)
                │   └─ byte[i*2] = int16[i] & 0xff
                │       byte[i*2+1] = (int16[i] >> 8) & 0xff
                │
                └─► onDataAvailable(uint8Data)
                    └─ 传递给 Coordinator
```

### 4.2 音频参数配置

| 参数 | 值 | 说明 |
|-----|-----|-----|
| 采样率 | 16000 Hz | 16kHz,语音识别标准 |
| 位深度 | 16 bit | PCM 16位 |
| 声道数 | 1 (单声道) | Mono |
| 编码格式 | PCM | 原始 PCM 数据 |
| 缓冲大小 | 4096 samples | 约 256ms @ 16kHz |
| 发送间隔 | 100 ms | 每 100ms 发送一批 |
| 数据量 | 3200 bytes/batch | 16000Hz * 16bit / 8 * 0.1s = 3200 |

### 4.3 音频发送流程

```
Coordinator.handleAudioData(data)
    │
    ├─ 场景判断
    │   │
    │   ├─ A. 活动会话正在录音
    │   │   └─► session.state === 'recording'
    │   │       │
    │   │       ├─ WS 已连接 (ws.canSend())
    │   │       │   └─► 直接发送
    │   │       │       └─ ws.sendAudio(data, isLast=false)
    │   │       │
    │   │       └─ WS 未就绪
    │   │           └─► 缓冲
    │   │               └─ session.audioBuffer.push(data)
    │   │
    │   └─ B. 有排队会话
    │       └─► 缓冲到最后一个排队会话
    │           └─ pendingSessions[last].audioBuffer.push(data)
    │
    └─► WSManager.sendAudio(audioData, isLast)
        │
        └─► createAudioOnlyRequest(audioData, isLast)
            │
            ├─ 可选: Gzip 压缩音频
            │
            └─ 组装二进制包
                ├─ Header (4 bytes)
                │   ├─ messageType: 0x01 (AudioOnly)
                │   ├─ messageFlags: isLast ? 0x01 : 0x00
                │   └─ compressionMethod: 0x1 (Gzip) 或 0x0
                │
                ├─ PayloadSize (4 bytes)
                │
                └─ Payload (音频数据)
```

## 五、ASR 结果接收与处理

### 5.1 消息接收流程

```
ws.onmessage(event)
    │
    └─► parseServerResponse(event.data)
        │
        ├─ 1. 解析协议头 (4 bytes)
        │   └─ 提取: messageType, messageFlags, compression
        │
        ├─ 2. 消息类型判断
        │   │
        │   ├─ ServerError (0x0f)
        │   │   └─► 解析错误码和错误信息
        │   │       └─ 返回: { error, code }
        │   │
        │   └─ FullServerResponse (0x09)
        │       └─► 继续解析
        │
        ├─ 3. 跳过序列号 (如果有)
        │   └─ messageFlags & WithSequence
        │       └─ offset += 4
        │
        ├─ 4. 读取 payload size (4 bytes)
        │   └─ payloadSize = big-endian 32-bit integer
        │
        ├─ 5. 提取并解压 payload
        │   ├─ payload = buffer.slice(offset, offset + payloadSize)
        │   └─ if compression === Gzip
        │       └─ payload = pako.ungzip(payload)
        │
        ├─ 6. 反序列化 JSON
        │   └─ result = JSON.parse(payload)
        │
        ├─ 7. 添加 is_last_package 标志
        │   └─ if (messageFlags & LastPackage)
        │       └─ result.is_last_package = true
        │
        └─► 返回 ASR 结果对象
```

### 5.2 ASR 结果结构

```typescript
{
  code: 20000000,              // 成功码
  message: "Success",          // 消息
  is_last_package: false,      // 是否最后一包
  result: {
    text: "识别的完整文本",     // 当前完整文本
    utterances: [               // 语句数组
      {
        text: "你好",           // 语句文本
        start_time: 0,          // 开始时间 (ms)
        end_time: 500,          // 结束时间 (ms)
        definite: true,         // 是否确定 (true) 或临时 (false)
        words: [...]            // 词级别信息 (可选)
      },
      {
        text: "世界",
        start_time: 500,
        end_time: 1000,
        definite: false,        // 临时结果,可能会改变
      }
    ]
  }
}
```

### 5.3 状态更新流程

```
Coordinator.handleASRMessage(result, sessionId)
    │
    ├─ 1. 会话验证
    │   └─ 只处理活动会话的消息
    │       └─ if (activeSession.id !== sessionId) return
    │
    └─► RecognitionStore.processResult(result)
        │
        ├─ 2. 错误检查
        │   ├─ code !== 20000000 → 错误
        │   └─ error 字段存在 → 错误
        │
        ├─ 3. 更新完整文本
        │   └─ if (result.result.text)
        │       └─ fullText = result.result.text
        │
        ├─ 4. 更新语句列表
        │   └─ if (result.result.utterances)
        │       ├─ 过滤空文本语句
        │       ├─ 去重检查 (start_time, end_time, text)
        │       └─ utterances = validUtterances
        │
        ├─ 5. 标记最终包
        │   └─ if (result.is_last_package)
        │       └─ lastPackageReceived = true
        │
        └─ 6. 更新时间戳
            └─ if (有变化)
                └─ lastOutputChangeAt = Date.now()
```

### 5.4 语句分类

```
utterances 数组分为两类:

┌─────────────────────────────────────────┐
│  definiteUtterances (确定语句)          │
│  - definite: true                       │
│  - 识别结果已稳定,不会再改变            │
│  - 用于提交和显示                       │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│  pendingUtterances (临时语句)           │
│  - definite: false                      │
│  - 可能随后续音频变化                   │
│  - 仅用于实时显示                       │
└─────────────────────────────────────────┘

LiveTranscribeView 的 watch:
  └─► watch(recognitionStore.definiteUtterances)
      └─ 新增确定语句时触发
          ├─ PRESS_TOGGLE 模式 → 累积缓冲
          └─ PRESS_HOLD 模式 → 累积缓冲
```

## 六、录音结束与终结流程

### 6.1 停止触发

```
用户松开按键 / 再次按下切换键
    │
    └─► coordinator.handleStop()
        │
        ├─ 1. 计算录音时长
        │   └─ duration = Date.now() - session.startTime
        │
        ├─ 2. 误触检测
        │   └─ if (duration < 120ms)
        │       └─► handleMistouch(session)
        │           ├─ 丢弃所有音频缓冲
        │           ├─ 不写入数据库
        │           ├─ session.state = 'completed'
        │           └─► 处理下一个排队会话
        │
        ├─ 3. 停止录音
        │   ├─ audioRecorder.stopRecording()
        │   │   ├─ 停止麦克风采集
        │   │   ├─ 关闭 AudioContext
        │   │   └─ 清空缓冲区
        │   │
        │   └─ stopSendingAudio()
        │       └─ 清除定时器
        │
        ├─ 4. 场景匹配 (如果启用)
        │   └─ if (shouldFetchActiveApp())
        │       └─► fetchActiveAppAndMatchScenario()
        │           ├─ 获取当前活动应用 (bundleId, name)
        │           ├─ LLM 匹配场景
        │           └─ 设置临时场景 (用于后处理)
        │
        └─ 5. 终结会话
            └─► finalizeSession(session)
```

### 6.2 会话终结流程

```
finalizeSession(session)
    │
    ├─ 1. 更新状态
    │   ├─ session.state = 'finalizing'
    │   └─ recognitionStore.status = 'processing'
    │
    ├─ 2. 发送剩余音频
    │   └─ for each chunk in session.audioBuffer
    │       └─ ws.sendAudio(chunk, isLast=false)
    │
    ├─ 3. 发送结束标记
    │   └─► ws.sendLastPacket()
    │       ├─ suppressReconnect = true (禁止重连)
    │       └─ sendAudio(empty, isLast=true)
    │           └─ 协议: messageFlags = 0x01 (LastPackage)
    │
    ├─ 4. 等待最终结果
    │   └─► ws.waitForFinal(timeout=3000ms)
    │       │
    │       ├─ 已收到 → 返回 finalResult
    │       │   └─ is_last_package === true
    │       │
    │       └─ 超时 → 返回 null
    │           └─ 使用已有的部分结果
    │
    ├─ 5. 关闭连接
    │   └─ ws.close()
    │       ├─ isManualClose = true
    │       └─ ws.readyState = CLOSED
    │
    └─ 6. 完成会话
        └─ session.state = 'completed'
```

### 6.3 最终结果处理

```
LiveTranscribeView.watch(lastPackageReceived)
    │
    ├─ 触发条件
    │   ├─ lastPackageReceived === true
    │   ├─ finalizeProcessed === false (防止重复)
    │   └─ mode !== 'PRESS_TOGGLE' (面板模式单独处理)
    │
    └─► finalizeWithPostprocess({ writeToInput: true })
        │
        ├─ 1. 获取完整文本
        │   └─ fullRawText = recognitionStore.fullText
        │       └─ 或 bufferedUtterances.map(u => u.text).join('')
        │
        ├─ 2. 后处理 (如果启用)
        │   └─► postprocessTextChunk(streamId, segment, rawText)
        │       ├─ 等待场景匹配完成
        │       │   └─ coordinator.waitForScenarioMatching()
        │       │
        │       ├─ 创建 LLM 客户端
        │       ├─ 构建 prompt (系统提示 + 热词 + 场景角色)
        │       ├─ 调用 LLM API
        │       │   └─ 标点优化、热词替换、格式调整
        │       │
        │       └─ 返回优化后的文本
        │
        ├─ 3. 更新 UI
        │   └─ recognitionStore.processResult({
        │       text: finalText,
        │       utterances: [...]
        │     })
        │
        ├─ 4. 保存到数据库
        │   └─► ensureRecordAndUpdateText(dbText, meta)
        │       ├─ 创建或更新数据库记录
        │       ├─ 保存原始文本 (rawText)
        │       ├─ 保存优化文本 (finalText)
        │       ├─ 保存场景信息 (scenarioId, scenarioName)
        │       └─ 保存时间指标 (postprocessTimeMs, vadConfirmTime)
        │
        ├─ 5. 写入输入框 (PRESS_HOLD 模式)
        │   └─ if (shouldWriteToInput)
        │       └─ electronAPI.inputWriter.write(text)
        │           └─ 模拟键盘输入到活动应用
        │
        └─ 6. 延迟隐藏 Toast (如果有后处理)
            └─ 延迟 100ms 显示最终文本
```

## 八、边界条件与异常处理

### 8.1 误触检测

```
目的: 防止用户误触快捷键产生无意义的录音记录

触发条件:
  └─ 录音时长 < 120ms (MISTOUCH_THRESHOLD_MS)

处理流程:
    │
    ├─ 1. 立即停止录音
    │   └─ audioRecorder.stopRecording()
    │
    ├─ 2. 丢弃所有数据
    │   ├─ session.audioBuffer = []
    │   └─ 不发送到 ASR
    │
    ├─ 3. 不写入数据库
    │   └─ recordId 保持为 null
    │
    ├─ 4. 不显示在 UI
    │   └─ recognitionStore.status = 'idle'
    │
    ├─ 5. 统计计数
    │   └─ mistouchCount++
    │
    └─ 6. 处理下一个排队会话
        └─► processNextSession()
```

### 8.2 连接失败处理

```
场景: WebSocket 连接超时或失败

WSManager.connect() 失败
    │
    ├─ 1. 捕获错误
    │   ├─ Connection timeout (10秒)
    │   ├─ Network error
    │   └─ Authentication error
    │
    ├─ 2. 更新状态
    │   ├─ state = 'disconnected'
    │   └─ recognitionStore.status = 'error'
    │
    ├─ 3. 显示错误信息
    │   └─ errorMessage = error.message
    │       └─ UI 显示红色错误提示
    │
    ├─ 4. 清理会话
    │   ├─ session.state = 'completed'
    │   ├─ 停止录音
    │   └─ 释放资源
    │
    └─ 5. 处理队列
        └─► processNextSession()
            └─ 尝试下一个会话 (可能成功)
```

### 8.3 认证失败处理

```
场景: AppKey 或 Token 无效

检查点:
    │
    ├─ A. 启动前检查 (checkCredentials)
    │   └─ if (!appid || !token)
    │       └─ errorMessage = "Please configure API credentials"
    │
    ├─ B. ASR 响应错误码
    │   └─ if (result.code !== 20000000)
    │       └─ errorMessage = result.message
    │           └─ 例: "Authentication failed"
    │
    └─ C. 服务器错误消息 (ServerError 0x0f)
        └─► parseServerResponse()
            └─ 返回: { error, code }
                └─ UI 显示错误

用户提示:
  └─ "Please configure API credentials in Settings"
      └─ 引导用户到设置页面
```

### 8.4 网络中断处理

```
场景: 录音中途网络断开

触发: ws.onclose (非手动关闭)
    │
    ├─ 1. 检测中断
    │   └─ state === 'connected' (录音中)
    │       └─ isManualClose === false
    │
    ├─ 2. 自动重连 (最多 3 次)
    │   └─► attemptReconnect()
    │       ├─ 等待 1 秒
    │       └─► connect(config)
    │           ├─ 成功 → 继续录音
    │           └─ 失败 → 重试 (最多 3 次)
    │
    ├─ 3. 重连失败
    │   └─ reconnectAttempts >= 3
    │       ├─ 停止录音
    │       ├─ 保存已有结果
    │       └─ 显示错误: "Connection lost"
    │
    └─ 4. 数据保护
        └─ 已缓冲的音频不丢失
            └─ session.audioBuffer 保留
                └─ 重连后继续发送
```

### 8.5 超长录音处理

```
场景: 用户长时间按住或忘记关闭

限制:
  └─ 无硬性时长限制 (依赖服务端)

队列保护:
  └─ MAX_PENDING_SESSIONS = 5
      └─ 防止无限排队

内存保护:
  └─ audioBuffer 持续发送
      └─ 不会无限积累

服务端限制:
  └─ ASR 服务可能有单次时长限制
      └─ 超时可能触发 is_last_package
```

### 8.6 快速连续按压

```
场景: 用户快速多次按下快捷键

处理策略:
    │
    ├─ A. 第一次按下
    │   └─ 创建 session-1
    │       └─ activeSession = session-1
    │           └─ 开始录音
    │
    ├─ B. 第二次按下 (session-1 还在录音)
    │   └─ 创建 session-2
    │       └─ 加入队列: pendingSessions.push(session-2)
    │           └─ 立即开始缓冲音频
    │
    ├─ C. 第三次按下
    │   └─ 创建 session-3
    │       └─ 继续排队
    │
    ├─ D. ... (最多 5 个排队)
    │
    └─ E. 第六次按下
        └─ 拒绝: "Session queue is full"
            └─ 不创建新会话

完成顺序:
  └─ session-1 完成 → 立即启动 session-2
      └─ session-2 完成 → 立即启动 session-3
          └─ ...依次执行
```

### 8.7 数据库写入失败

```
场景: 磁盘满、权限不足、数据库损坏

处理流程:
    │
    ├─ 1. 尝试创建/更新
    │   └─► ensureRecordAndUpdateText()
    │       └─ result.success === false
    │
    ├─ 2. 日志记录
    │   └─ logger.error('Failed to update record:', error)
    │
    ├─ 3. 验证写入 (面板模式)
    │   └─► 读回验证
    │       └─ getRecordById(recordId)
    │           ├─ 成功 → 继续
    │           └─ 失败 → 显示错误
    │
    ├─ 4. 用户提示
    │   └─ panelStatusText = "保存失败,请稍后在历史记录查看"
    │
    └─ 5. 数据保护
        └─ 文本已在 committedText 中
            └─ 用户可从 UI 复制
                └─ 手动保存
```

### 8.8 后处理超时

```
场景: LLM API 超时或失败

配置:
  └─ timeoutMs: 10000 (默认 10 秒)

处理:
    │
    └─► postprocessTextChunk()
        │
        ├─ 超时/失败
        │   └─ return { ok: false, reason: 'error' }
        │
        └─ 回退策略
            └─ 使用原始文本 (rawText)
                ├─ 保存到数据库
                ├─ 写入输入框
                └─ 不影响核心功能

日志:
  └─ "Postprocess skipped/failed, using raw text"
```

### 8.9 浏览器兼容性

```
检查点:
    │
    ├─ 1. Web Audio API
    │   └─ window.AudioContext || window.webkitAudioContext
    │       └─ 不支持 → 错误提示
    │
    ├─ 2. WebSocket
    │   └─ 所有现代浏览器支持
    │
    ├─ 3. ArrayBuffer / Uint8Array
    │   └─ 所有现代浏览器支持
    │
    └─ 4. Electron API
        └─ 仅在 Electron 环境可用
            └─ window.electronAPI 存在性检查
```

## 九、性能优化策略

### 9.1 音频处理优化

```
1. 采样策略
   └─ 音量计算时采样: 每 4 个样本取 1 个
      └─ 减少 75% 计算量
      └─ 对结果影响可忽略

2. 批量发送
   └─ 不逐个采样发送,而是批量 (100ms)
      └─ 减少网络开销
      └─ 提高传输效率

3. 数据压缩
   └─ Gzip 压缩音频和配置
      └─ 压缩比: 约 50-70%
      └─ 减少带宽消耗
```

### 9.2 UI 更新优化

```
1. 防抖更新
   └─ lastOutputChangeAt 时间戳
      └─ 仅在内容真正变化时触发
      └─ 避免无效渲染

2. 计算缓存
   └─ computed() 属性
      ├─ definiteText
      ├─ pendingText
      └─ 仅在依赖变化时重新计算

3. 列表去重
   └─ 基于 start_time + end_time 去重
      └─ 避免重复渲染相同语句
```

### 9.3 内存管理

```
1. 及时清理
   └─ 音频缓冲发送后立即清空
      └─ audioBuffer = []

2. 连接复用
   └─ 每个会话独立 WebSocket
      └─ 完成后立即关闭
      └─ 不保持长连接

3. 状态重置
   └─ streamId 变化时触发完整重置
      └─ 清空所有累积数据
```

### 9.4 并发控制

```
1. 会话队列
   └─ 同时只有 1 个活动会话
      └─ 其他排队等待
      └─ 防止资源竞争

2. 提交互斥
   └─ runCommitExclusive() 串行化
      └─ commitChain Promise 链
      └─ 防止数据库写冲突

3. 后处理队列
   └─ PostprocessQueue
      └─ 相同 streamId 串行
      └─ 不同 streamId 可并发
```

## 十四、核心设计原则

### 14.1 单一职责

- **AudioRecorder**: 仅负责音频采集和预处理
- **WSManager**: 仅负责 WebSocket 连接管理
- **Coordinator**: 负责会话生命周期和协调
- **RecognitionStore**: 负责状态和结果管理
- **LiveTranscribeView**: 负责 UI 和业务编排

### 14.2 错误隔离

- 每个会话独立,一个失败不影响其他
- 网络错误不影响本地录音
- 后处理失败回退到原始文本
- 数据库失败不丢失文本 (UI 可复制)

### 14.3 渐进增强

- 核心功能: 录音 + ASR 识别
- 增强功能: 后处理优化 (可选)
- 高级功能: 场景匹配 (可选)
- 关闭增强功能不影响核心

### 14.4 用户友好

- 误触检测: 避免无意义记录
- 队列机制: 快速连按不丢失
- 自动重连: 网络波动自动恢复
- 错误提示: 清晰的错误信息

### 14.5 性能优先

- 零延迟启动: 按键立即响应
- 流式处理: 实时显示结果
- 批量发送: 减少网络开销
- 及时清理: 避免内存泄漏

## 十五、常见问题排查

### Q1: 无法开始录音

```
排查步骤:
  1. 检查麦克风权限
     └─ 浏览器/系统设置
  
  2. 检查 API 配置
     └─ Settings → AppKey & Token
  
  3. 查看错误信息
     └─ errorMessage 显示
  
  4. 检查浏览器兼容性
     └─ AudioContext 支持
```

### Q2: 识别结果不准确

```
可能原因:
  1. 麦克风质量差
  2. 环境噪音大
  3. 说话不清晰
  4. 网络延迟高

改善方法:
  ├─ 使用高质量麦克风
  ├─ 安静环境录音
  ├─ 清晰发音,适当停顿
  └─ 检查网络连接
```

### Q3: 后处理失败

```
排查步骤:
  1. 检查 LLM API Key
     └─ Settings → LLM Postprocess
  
  2. 检查网络连接
     └─ API 可访问性
  
  3. 查看超时设置
     └─ timeoutMs 配置
  
  4. 查看日志
     └─ Console 错误信息

回退机制:
  └─ 自动使用原始文本
      └─ 不影响核心功能
```

### Q4: 面板不自动关闭

```
可能原因:
  1. 未收到最终包
     └─ 网络问题
  
  2. 后处理未完成
     └─ LLM 超时
  
  3. 数据库写入失败
     └─ 权限/空间问题

解决方法:
  └─ 手动复制文本
      └─ 面板支持选择复制
```

### Q5: 会话队列满

```
错误提示:
  └─ "Session queue is full"

原因:
  └─ 快速按压超过 5 次

解决方法:
  1. 等待当前会话完成
  2. 避免快速连续按压
  3. 检查是否有卡死的会话
```
